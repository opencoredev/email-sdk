An adapter is a plain object implementing `EmailProvider`: one routing name, one `send` function that maps a normalized [message](/docs/reference/message) to one provider API. This page is the type contract; the walkthrough is [Create an adapter](/docs/guides/authoring/create-adapter).

## `EmailProvider`

```ts
type EmailProvider<TRaw = unknown> = {
  name: string;
  send(message: EmailMessage, context: EmailProviderContext): MaybePromise<EmailProviderResponse>;
  sendBulk?(message: EmailMessage, context: EmailProviderContext): MaybePromise<EmailProviderResponse>;
  raw?: TRaw;
};
```

```ts
import { EmailProviderError } from "@opencoredev/email-sdk";
import type { EmailProvider } from "@opencoredev/email-sdk";

export const acmeMail: EmailProvider = {
  name: "acme-mail",
  async send(message, context) {
    const response = await fetch("https://api.acme-mail.example/send", {
      method: "POST",
      signal: context.signal,
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify(toAcmePayload(message)),
    });

    if (!response.ok) {
      throw new EmailProviderError(`acme-mail failed with HTTP ${response.status}.`, {
        provider: "acme-mail",
        status: response.status,
        retryable: response.status === 429 || response.status >= 500,
      });
    }

    const body = await response.json();
    return { provider: "acme-mail", id: body.id, raw: body };
  },
};
```

## Response contract

Return an [`EmailProviderResponse`](/docs/reference/client#response). Set `provider` to the adapter's routing name (the client fills it in if left empty), map the provider's message id into `id`/`messageId`, and keep the untouched body in `raw` for debugging.

## Error contract

Throw `EmailProviderError` for provider failures and classify retryability so the client's [retry layer](/docs/concepts/fallbacks-and-retries) works:

- Set `status` to the HTTP status when there is one.
- Set `retryable: true` for transient failures. The SDK's own adapters use `isRetryableStatus`: `408`, `409`, `425`, `429`, and any `5xx`.
- Put the parsed error body in `details`.

Anything else you throw is normalized by the client via `toProviderError`: plain `Error`s become non-retryable `EmailProviderError`s — except network-style failures (`ECONNRESET`-family codes, `fetch failed`, timeouts), which are marked retryable. `AbortError` is never retried. See the [errors reference](/docs/reference/errors) for the full taxonomy.

## Fail fast on unsupported fields

Adapters must reject message fields the provider cannot represent — before any request, never a silent drop. Built-in adapters throw `EmailValidationError` with this exact shape:

```txt
<adapter> does not support these EmailMessage fields: tags, metadata.
```

Do the same in custom adapters: check `cc`, `bcc`, `replyTo`, `headers`, `attachments`, `tags`, and `metadata` against what your provider supports, and throw for the rest. This is what keeps [fallback routes](/docs/adapters/field-support#choosing-compatible-routes) honest.

## Native batch sends (`sendBulk`)

`sendBulk` is optional. Implement it only when your provider can personalize a whole `to` list in **one API call** — substituting the `%recipient.key%` tokens of [`recipientVariables`](/docs/reference/message#recipient-variables) server-side. Currently [Mailgun](/docs/adapters/mailgun) and [SendGrid](/docs/adapters/sendgrid) do.

The client routes a message to `sendBulk` instead of `send` when — and only when — the message carries `recipientVariables` **and** the adapter defines `sendBulk`. The message reaching `sendBulk` is the original, unexpanded one: the full `to` list, `recipientVariables` still attached, and `%recipient.key%` tokens still literal in `subject`, `html`, and `text`. Your job is to hand all of that to the provider's batch endpoint. Return one [`EmailProviderResponse`](/docs/reference/client#response) for the whole batch.

`sendBulk` runs through the same [retry layer](/docs/concepts/fallbacks-and-retries) as `send`, receives the same [`EmailProviderContext`](#emailprovidercontext), and is invoked as a method on the adapter — so a class-based adapter that reads `this` inside `sendBulk` keeps working. Throw `EmailProviderError` the same way; a batch that exhausts retries falls through to the next route like any other failure.

**Fail fast past the provider's batch cap.** Providers bound a single batch call — Mailgun and SendGrid both at 1000 recipients. Guard it with the shared `assertMaxItems` helper (`src/utils.ts`) so an over-limit batch throws `EmailValidationError` before any request, rather than being silently truncated by the provider:

```ts
assertMaxItems("acme-mail", "recipient", arrayify(message.to), 1000);
```

The built-in adapters implement this by branching inside their existing payload builder on `hasRecipientVariables(message)` and pointing `sendBulk` at the same `send` function:

```ts
const provider = jsonProvider({
  name: "acme-mail",
  // ...buildPayload emits per-recipient personalizations when hasRecipientVariables(message)
});

return { ...provider, sendBulk: provider.send };
```

**If you omit `sendBulk`, `recipientVariables` still works** — the client falls back to one rendered `send` per recipient, substituting tokens client-side (an unknown token is left intact, matching native providers). Each recipient is a real send: `afterSend` fires per delivered recipient and `onError` per failed one, any idempotency key is suffixed `<key>:<address>`, and results aggregate into the batch response's `accepted`/`rejected`. The adapter only counts as failed — triggering the next fallback route — when every recipient fails. So implement `sendBulk` purely as a performance optimization (one request instead of N), never to *enable* the feature. Because the fallback substitutes with the same `[\w-]` token grammar the validation layer enforces, native and fallback routes personalize identically.

## Internal helpers

The built-in adapters share two internal modules in the SDK repository — useful as reference, or directly if you contribute an adapter upstream (they are not public package exports):

- `src/http.ts` — `jsonProvider({ name, baseUrl, endpoint, headers, buildPayload, parseResponse? })` wraps the whole JSON-API pattern: fetch with `context.signal`, error-body parsing, `EmailProviderError` with `isRetryableStatus`, and response normalization.
- `src/payloads.ts` — address and attachment mapping helpers (`apiAddresses`, `stringAddresses`, `base64Attachments`, `commonHeadersObject`, …) used across the 20 adapters.

Community packages should copy these patterns rather than import them.

## Packaging as a plugin

Ship a reusable adapter as an `EmailPlugin` so users mount it with `plugins: [...]`:

```ts
import type { EmailPlugin } from "@opencoredev/email-sdk";

export function acmeMailPlugin(options: AcmeMailOptions): EmailPlugin {
  return {
    id: "acme-mail",
    adapters: [createAcmeMail(options)],
  };
}
```

`adapters` must be synchronous — an array, or a function of the plugin context that returns one. A plugin that returns a Promise throws `EmailValidationError` at client creation, as do duplicate plugin ids and duplicate adapter names.

Export both the plain adapter factory and the plugin from your package root, then follow [Publish a community adapter](/docs/guides/authoring/publish-community-adapter) to list it in the [community registry](/docs/reference/community-registry).
