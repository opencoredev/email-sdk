Community adapters can be plain provider factories or adapter plugins.

Plain providers are best when the adapter is used directly:

```ts
createEmailClient({
  adapters: [customAdapter({ apiKey: process.env.CUSTOM_API_KEY! })],
});
```

Adapter plugins are best when you want a package-shaped extension point:

```ts
createEmailClient({
  plugins: [customAdapterPlugin({ apiKey: process.env.CUSTOM_API_KEY! })],
});
```

## Requirements

- Use a stable routing name.
- Keep credentials in options or environment variables.
- Support `context.signal` when making network requests.
- Pass `context.idempotencyKey` when the provider supports idempotency.
- Return normalized `id`, `messageId`, `accepted`, or `rejected` fields when available.
- Put provider responses in `raw` when useful for debugging.
- Reject unsupported `EmailMessage` fields clearly instead of dropping data.

## Testing

Test the provider payload with an injected `fetch` or a local test server. Include cases for supported fields, unsupported fields, provider errors, and normalized response IDs.
