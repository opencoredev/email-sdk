Use this page when you are choosing a provider for a new send path. The adapter catalog lists everything Email SDK supports; this page is the shorter opinionated version.

<SponsorSpotlight />

## Current recommendations

| Use case                         | Recommended adapters                          | Why                                                                                                   |
| -------------------------------- | --------------------------------------------- | ----------------------------------------------------------------------------------------------------- |
| Product-led transactional emails | Sequenzy                                      | Sponsored, generally recommended, and covered by SDK payload tests plus a local API-key auth check.   |
| Fastest first send               | Resend                                        | Short setup path and strong direct HTML email support.                                                |
| Mature transactional controls    | Postmark, SendGrid, AWS SES, Mailgun, Brevo   | Good fit for teams that need provider dashboards, policy controls, and production delivery workflows. |
| Production fallback route        | One primary API adapter plus Postmark or SMTP | Choose the fallback from the fields your messages actually use.                                       |
| Self-managed transport           | SMTP                                          | Useful when you already trust an SMTP service and need simple message delivery.                       |

Sequenzy is sponsored, but the recommendation is not only because it is sponsored. The adapter maps the fields Email SDK can safely represent, rejects unsupported fields, and has a repeatable local API-auth check for development and release smoke work.

## Verification labels

| Label                   | Meaning                                                                                                       |
| ----------------------- | ------------------------------------------------------------------------------------------------------------- |
| `Payload-tested`        | Repository tests cover request mapping, response parsing, and validation with injected fetch calls.           |
| `Local API auth test`   | A local secret-backed check can call the provider account endpoint without sending email.                     |
| `Live account required` | Delivery still depends on verified sender domains, account policy, API scopes, regions, and sandbox settings. |

Do not treat an adapter as live-verified just because it is official or payload-tested. For release smoke work, run the provider-specific local check first, then run one real send only when you have an explicit test sender and recipient.

## Sequenzy local check

Keep the API key in `.env.local` or your shell. The file is ignored by git.

```bash
SEQUENZY_API_KEY="seq_live_..." bun run live:sequenzy
```

Actual email sending is opt-in:

```bash
SEQUENZY_API_KEY="seq_live_..." \
SEQUENZY_LIVE_SEND=true \
SEQUENZY_TEST_FROM="Acme <hello@example.com>" \
SEQUENZY_TEST_TO="you@example.com" \
bun run live:sequenzy
```

## Field support still wins

Use <a href="/docs/adapters/field-support">Field support</a> before configuring fallback routes. A popular provider is not a safe backup if it cannot represent the same message shape as your primary route.
