Plugins package reusable Email SDK behavior.

Use a plugin when you want to share an adapter setup, add defaults before messages are validated, observe sends, capture test events, or expose a small helper on the client.

```ts
import { createEmailClient } from "email-sdk";
import { defaultsPlugin } from "email-sdk/plugins/defaults";
import { observabilityPlugin } from "email-sdk/plugins/observability";
import { resend } from "email-sdk/resend";

const email = createEmailClient({
  adapters: [resend({ apiKey: process.env.RESEND_API_KEY! })],
  plugins: [
    defaultsPlugin({
      headers: { "X-App": "billing" },
      sendMetadata: { service: "billing" },
    }),
    observabilityPlugin({
      log(event) {
        console.log(event.type, event.provider);
      },
    }),
  ],
});
```

## What plugins can do

- Register adapters.
- Run `beforeSend` middleware before message validation.
- Observe successful sends, retries, and errors.
- Add small client extensions.
- Compose with user `hooks`.

## What plugins do not do

Email SDK plugins do not add databases, migrations, HTTP endpoints, route middleware, rate limits, or server/client plugin pairs. Those ideas are useful in larger frameworks, but Email SDK keeps plugins focused on sending email.

## Order

Email SDK applies plugins in the order you pass them.

1. Direct `adapters` register first.
2. Plugin adapters register next.
3. Duplicate plugin IDs fail.
4. Duplicate adapter names fail.
5. `beforeSend` middleware runs before message validation.
6. Plugin hooks run before user `hooks`.

This lets application code keep using the same `send` call while plugins add reusable behavior around it.

Built-in plugins use stable IDs like `defaults`, `observability`, and `capture`. Pass a custom `id` when you need more than one instance of the same plugin type. If the plugin extends the client, also give each instance a distinct extension key. For example, `capturePlugin({ id: "capture:audit", clientKey: "auditCapture" })` exposes `email.auditCapture`.
